## Project Description

This project compares the **code generation** capabilities of three language models belonging to
different architectural families: a frontier **dense LLM** (Claude Opus 4.7), a **Mixture of
Experts** (DeepSeek V4 Pro) and a **Small Language Model** (Ministral 8B). The goal is not to
determine which commercial product is better, but to observe how the three **architectures** behave
on the same task, regardless of differences in training data, parameter count and the optimization
choices of the individual providers. The three architectures are described in the
[Architectures Compared](#architectures-compared-llm-moe-and-slm) section.

The evaluation uses **five benchmarks**: four of them (HumanEval, MBPP, DS-1000 and Plot2Code) focus
on **Python**, while **MultiPL-E** extends the comparison to **24 programming languages**. To keep
computational costs down, of those 24 languages only **15** are actually run — the ones with a
reliable runtime available on the test machine (the full list is in the *Benchmark: MultiPL-E*
section).

The objective is to measure how well the models solve the programming problems they are given **on
the first attempt**. To this end, two complementary metrics are used:

- **pass@k**, the functional correctness metric: it estimates the probability that at least one of
  the k solutions generated by the model passes all of the problem's unit tests. In this work
  **k = 1** (pass@1): the model gets a single attempt, the strictest condition and the closest to
  real-world use.
- **CodeBLEU**, a structural metric designed specifically for code: it compares the code generated
  by the model with the benchmark's reference solution by combining four components (n-gram overlap,
  n-grams weighted on the language's keywords, abstract syntax tree similarity and data-flow
  similarity between variables). The comparison happens regardless of whether the generated code
  works or not: it measures how much the solution **resembles** the reference, not whether it is
  correct. The detailed workings are described in the *Evaluation Metrics* section.

For each run, **tokens and cost** are also tracked, along with the **error category** (SyntaxError,
AssertionError, TimeoutError, etc.) for unsolved problems.

Two notes on the less conventional benchmarks:

> **Plot2Code** (`TencentARC/Plot2Code`) is the **visualization** benchmark: given the textual
> description of a figure, the model generates the **matplotlib script**; correctness is the
> **successful rendering** (pass@1), accompanied by CodeBLEU and a composite **image similarity**.
>
> **MultiPL-E** (`nuprl/MultiPL-E`, HumanEval set) is the **multilingual** benchmark: the same
> HumanEval problems translated into the various languages, gathered into a single benchmark with a
> `language` column and one result file. It does not provide a gold solution, so the metric is
> **pass@1 only** (no CodeBLEU). pass@1 requires the language runtime to be installed; to avoid
> wasting API calls, the pipeline detects the runtimes present and **generates only for those
> languages** (the rest result in `RuntimeMissing`).

## Architectures Compared: LLM, MoE and SLM

All three models share the same foundation: the **decoder-only Transformer**, pre-trained to predict
the next token over large corpora of text and code, and subsequently refined in post-training
(instruction tuning and reinforcement learning from human feedback). What distinguishes the three
families is **how the parameters are organized and how many of them are activated** to generate each
token.

### Dense LLM (Claude Opus 4.7)

In a **dense** model every token passes through **all** of the network's parameters: in each
Transformer block both the attention and the feed-forward layer take part in computing every
generated token. It is the architecture with the highest potential capacity, because the entire
network contributes to every prediction, but also the most expensive, since inference cost grows
with the total size of the model. Frontier models in this family are trained on corpora on the order
of **tens of trillions of tokens** (text, code and synthetic data); Anthropic does not disclose the
parameter count or the exact composition of Claude Opus 4.7's training data.

### Mixture of Experts (DeepSeek V4 Pro)

In a **MoE** model the feed-forward layer of each block is replaced by a set of **experts**
(independent copies of the layer) and a learned **router** that, token by token, routes the
computation only to a small subset of experts. DeepSeek V4 Pro reports **284 billion total
parameters, of which about 13 billion are active** per token: the overall knowledge is that of a
large model, the inference cost that of a medium-small one. The price is paid during training: the
router must be balanced with auxiliary loss functions, to prevent a few experts from monopolizing
the traffic, and the model still requires enough memory to host all of the experts, including the
inactive ones.

### Small Language Model (Ministral 8B)

An **SLM** is a small dense model (here **8 billion parameters**), designed to run with few
resources: edge devices, local execution, low latency. Unable to rely on scale, its quality depends
critically on the **care taken with the training data**: aggressive corpus filtering, high-quality
synthetic data and knowledge distillation from larger models. It is the architecture with the most
extreme cost/performance ratio, but with a structurally lower capacity ceiling, as the results of
this benchmark clearly show.

### Visual comparison of the three architectures

The three diagrams below, side by side, highlight the key difference. On the left the **dense**
GPT-style model: a stack of Transformer blocks in which each token passes through the single
feed-forward layer of each block, hence all the parameters. In the center the **MoE** block of

in green) and the router activates only a small fraction of them for each token. On the right the

### Installing dependencies

```bash
pip install -r requirements.txt
```

> **⚠️ CodeBLEU / tree-sitter note (important on Python 3.14).**
> `codebleu` pins `tree-sitter<0.23`, which **has no wheel for cp314** and fails to build from
> source. If `pip install -r requirements.txt` errors out on `tree-sitter`, install in two steps:
> ```bash
> pip install tree-sitter tree-sitter-python
> pip install codebleu --no-deps
> ```
> At runtime `codebleu` 0.7 works correctly with `tree-sitter` 0.25 despite the pin.

| Library | Purpose |
|----------|-------|
| `anthropic` | Official SDK to query Claude Opus 4.7 |
| `datasets` + `huggingface-hub` | Loading HumanEval from Hugging Face |
| `codebleu` + `tree-sitter` + `tree-sitter-python` | CodeBLEU structural metric (AST + data-flow) |
| `openpyxl` | Export of the detail to Excel (`.xlsx`) |
| `typer` | CLI: builds the commands from type hints and generates the `--help` (pulls in `click`) |
| `rich` | CLI interface: formatted `--help`, tables, progress bar, panels |
| `python-dotenv` | Loading the API key from a `.env` file |
| `pandas`, `numpy`, `scipy`, `scikit-learn`, `matplotlib`, `torch` | **(DS-1000 only)** runtime to execute solutions that import these libraries |

> **⚠️ Libraries for DS-1000.** Unlike HumanEval/MBPP (stdlib only), **DS-1000 executes code that
> imports data-science libraries**, which are **not** in `requirements.txt` (only the core
> dependencies live there). To run DS-1000, install them separately:
> ```bash
> pip install pandas numpy scipy scikit-learn matplotlib torch
> ```
> The pipeline **automatically detects** which ones are installed and **runs only the problems of
> the libraries present**, skipping the others with a warning (no silent truncation: see *Benchmark:
> DS-1000*). On **Python 3.14 TensorFlow does not yet have a wheel**, so its **45 problems** are
> excluded and pass@1 is reported over **955/1000**.

> **⚠️ Dependencies for Plot2Code.** Plot2Code renders figures: it needs `matplotlib` and `Pillow`
> (not in `requirements.txt`). For the text-only component of the image similarity you also need
> `pytesseract` **+** the Tesseract engine installed separately; without it, SSIM and color are
> still computed.
> ```bash
> pip install matplotlib pillow
> ```

> **⚠️ Toolchain for MultiPL-E (multilingual).** MultiPL-E's `pass@1` requires the **language
> runtime** installed (Node, JDK, Go, Rust, …): without it, the executor marks `RuntimeMissing` and
> moves on. The full list of languages with the **`winget` commands** to install them (and a
> copy-paste "quick install" block) is in the *Benchmark: MultiPL-E* section.

### API Key configuration

The benchmark queries each model through the API of its respective provider: **Anthropic** for
Claude, **DeepSeek** for DeepSeek V4 Pro and **Mistral** for Ministral 8B. Set the API keys of the
providers you want to test before starting, or place them in a `.env` file in the project root:

```bash
ANTHROPIC_API_KEY=your-api-key
DEEPSEEK_API_KEY=your-api-key
MISTRAL_API_KEY=your-api-key
```

> Only the **models whose API key is set** are run; the others are skipped with a warning. So the
> key of **at least one** provider is enough (you can run just one of the three models); with all
> three keys you get the full LLM vs MoE vs SLM comparison. The `model_id`s and prices should be
> verified on the providers' consoles (they change often); to select a subset, repeat the
> `--models` option (e.g. `--models claude-opus-4.7 --models deepseek-v4-pro`).

Or as environment variables:

On Windows (PowerShell):

```powershell
$env:ANTHROPIC_API_KEY="your-api-key"
$env:DEEPSEEK_API_KEY="your-api-key"
$env:MISTRAL_API_KEY="your-api-key"
```

On Linux/macOS:

```bash
export ANTHROPIC_API_KEY="your-api-key"
export DEEPSEEK_API_KEY="your-api-key"
export MISTRAL_API_KEY="your-api-key"
```

> You can get an API key from the providers' consoles:
> [console.anthropic.com](https://console.anthropic.com) (Claude),
> [platform.deepseek.com](https://platform.deepseek.com) (DeepSeek) and
> [console.mistral.ai](https://console.mistral.ai) (Mistral).

## How to Use

The entrypoint is `cli.py` (equivalent to `python -m model.claude`). The CLI is built with
**Typer**: the `--help` is formatted by **Rich** (options grouped into panels) and every option also
has a **short form** (`-b`, `-n`, `-m`, `-t`).

```bash
python cli.py
python cli.py --benchmark mbpp
python cli.py -b humaneval
python cli.py --benchmark ds1000
python cli.py --benchmark ds1000 --libraries Pandas --libraries Numpy
python cli.py --benchmark plot2code
python cli.py --benchmark multipl-e
python cli.py -b multipl-e -n 3
python cli.py --benchmark mbpp -n 5
python cli.py --models claude-opus-4.7 --models deepseek-v4-pro
python cli.py --benchmark mbpp --fresh
python cli.py --list
```

The meaning of each option (with its short form and default value) is summarized in the table below;
for MultiPL-E the `-n` limit is intended **per language**.

> **Multi-value options (Typer).** To pass multiple values, **repeat the option**:
> `--models a --models b`, `--libraries Pandas --libraries Numpy` (no longer space-separated).

If you omit `--benchmark`, a menu is shown at startup to let you choose (Enter = HumanEval). The
results of each benchmark end up in a **dedicated subfolder**: `results/humaneval/`, `results/mbpp/`
or `results/ds1000/` (see *Results Output*).

| Flag | Default | Description |
|------|---------|-------------|
| `--benchmark` / `-b` `NAME` | *asked at runtime* | Which benchmark: `humaneval` / `mbpp` / `ds1000` / `plot2code` / `multipl-e`. If omitted, interactive menu |
| `--limit` / `-n` `N` | all (164 HumanEval / 500 MBPP / 1000 DS-1000 / 368 Plot2Code / ~161×24 MultiPL-E) | Number of problems (useful for cheap tests). For MultiPL-E it is **per language** |
| `--libraries` `LIB` | all installed | **(DS-1000 only)** restricts to the listed libraries; **repeat** for more (e.g. `--libraries Pandas --libraries Numpy`, case-insensitive). Uninstalled ones are skipped with a warning |
| `--models` / `-m` `KEY` | all (with API key) | Model keys to test; **repeat** for more (e.g. `-m claude-opus-4.7 -m deepseek-v4-pro`) |
| `--timeout` / `-t` `SEC` | *per benchmark*: 10 HumanEval/MBPP · 60 DS-1000/Plot2Code · 30 MultiPL-E | Timeout in seconds for the execution of each test |
| `--fresh` | off | Ignores the checkpoint and restarts from scratch (deletes the `.jsonl`) |
| `--list` | n/a | Lists the configured models and exits |
| `--help` / `-h` | n/a | Shows the help (formatted with Rich) and exits |

> ⚠️ Do not run the files inside `model/` directly (e.g. `python model/pipeline.py`): they use
> relative imports and would error out. Use `python cli.py` or `python -m model.claude`.

### Inference configuration

The generation parameters live in `model/config.py`:

| Parameter | Value | Notes |
|-----------|--------|------|
| `CLAUDE_EFFORT` | `low` | `low\|medium\|high\|xhigh\|max`; raise it to measure maximum capacity |
| `CLAUDE_THINKING` | `disabled` | `disabled\|adaptive` |
| `MAX_TOKENS` | `2048` | headroom for signature + imports + function body |

> **⚠️ Opus 4.7 API parameters.** `temperature`/`top_p`/`top_k` are **removed** (sending them gives
> a 400 error). Behavior is controlled via `output_config={"effort": …}` and `thinking={"type":
> …}`. Prices: **$5 / 1M input tokens**, **$25 / 1M output tokens**. A full run costs about
> **$0.85**.

> **Comparison models (DeepSeek, Mistral).** They use the **OpenAI-compatible** API (same `openai`
> client, only `base_url` and key change) and **support `temperature`**, set to `0.0` in
> `config.py` for determinism. They have no `effort`/`thinking`: their behavior is driven solely by
> `SYSTEM_PROMPT` + DSP, **identical for all models** (fair comparison). They cost much less than
> Claude (see *Models compared*).

Loading the dataset in `model/humaneval.py` (164 problems):

```python
from datasets import load_dataset
ds = load_dataset("openai/openai_humaneval", split="test")
```

HumanEval is an OpenAI benchmark made up of **164 hand-written programming problems**, similar to
software interview questions. Each problem includes a **function signature**, a **docstring**
describing the expected behavior and a set of **unit tests** (7.7 per problem on average).

---

### Dataset Structure

| Column | Type | Description |
|---------|------|-------------|
| `task_id` | `string` | Unique identifier (e.g. `"HumanEval/0"`) |
| `prompt` | `string` | Imports + function signature + docstring with the instructions |
| `canonical_solution` | `string` | **Only the body** of the reference solution (indented) |
| `test` | `string` | Test function with the unit tests to validate the code |
| `entry_point` | `string` | Name of the function to implement |

---

### Derived column: `codice_completo`

In HumanEval the `prompt` contains imports + signature + docstring, while `canonical_solution`
contains **only the body** of the function. At load time (`humaneval.py`) we build a derived column:

```
codice_completo = prompt + canonical_solution
```

i.e. the **complete, executable reference function** (imports + signature + docstring + body). This
is the reference used for **CodeBLEU**: the model also generates the complete function, so the
"complete vs complete" comparison is fair (see below).

#### Example (problem `HumanEval/0`)

`prompt` (imports + signature + docstring):

```python
from typing import List

def has_close_elements(numbers: List[float], threshold: float) -> bool:
    """ Check if in given list of numbers, are any two numbers closer to each
    other than given threshold.
    >>> has_close_elements([1.0, 2.0, 3.0], 0.5)
    False
    """
```

`canonical_solution` (body only, indented):

```python
    for idx, elem in enumerate(numbers):
        for idx2, elem2 in enumerate(numbers):
            if idx != idx2 and abs(elem - elem2) < threshold:
                return True
    return False
```

The concatenation of the two (`codice_completo = prompt + canonical_solution`) is the complete
reference function.

---

### Windows note on loading

The dataset is downloaded once and saved to `Benchmark/humaneval/` with `save_to_disk`, then read
back from there on subsequent runs. We do **not** use `load_dataset(cache_dir=Benchmark)` because
Hugging Face's lock file embeds the entire absolute path and, on deep paths, exceeds Windows' 260
character limit.

**MBPP** (*Mostly Basic Python Problems*) is a second code generation benchmark: problems described
in **natural language** (not from signature+docstring like HumanEval), each with a **reference
solution** and a list of `assert` statements to pass.

---

### Example (problem `task_id 12`)

`text` (natural-language description, the input to the model):

```text
Write a function to sort a given matrix in ascending order according to the sum of its rows.
```

`code` (reference solution: already the complete function, `def` + body):

```python
def sort_matrix(M):
    result = sorted(M, key=sum)
    return result
```

`test_list` (the `assert` statements the code must pass):

```python
assert sort_matrix([[1, 2, 3], [2, 4, 5], [1, 1, 1]]) == [[1, 1, 1], [1, 2, 3], [2, 4, 5]]
```

> Unlike HumanEval there is no signature to start from: the prompt infers the function's name and
> signature **from the tests** (see *Prompting: Directional Stimulus*).

---

### Which part of MBPP we use

The "full" dataset (974 problems) is split **by `task_id` ranges** (the paper's convention):

| Split | `task_id` | Problems | Purpose |
|---|---|---:|---|
| `prompt` | 1-10 | 10 | few-shot examples to insert in the prompt |
| **`test`** | **11-510** | **500** | **evaluation ← we use this** |
| `validation` | 511-600 | 90 | hyperparameter tuning |
| `train` | 601-974 | 374 | fine-tuning |

We use **only the `test` split (500 problems, `task_id` 11-510)**: it is the held-out evaluation set
on which papers report pass@1. The other splits are not needed for our case (zero-shot, no training,
no systematic tuning).

> **Robustness to the dataset version.** The split *names* can vary between dataset revisions, but
> the `task_id` is the **stable** identity of each problem. For this reason `mbpp.py` does not trust
> the split label: it loads the "full" config, merges all the rows and selects the test **by
> filtering on `task_id` 11-510**. This way we always get the same 500 problems, however the splits
> are labeled in the downloaded copy.

---

### Why `full` and NOT `sanitized`

MBPP has two configurations: **`full`** (974 problems) and **`sanitized`** (427, a manually verified
subset that **removes** problems with ambiguous/ungrammatical descriptions and **polishes** the
wording of others). The natural choice for a "clean" comparison would be `sanitized`, but here we
**deliberately use `full`**, for a precise reason:

- **We also want to test robustness to badly written prompts.** Some of the `full` problems have
  imperfect descriptions (poor grammar, ambiguous phrasing). Evaluating Claude *also* on these
  measures a real and interesting capability (**can it handle a poorly worded specification?**) that
  `sanitized`, by cleaning everything up, would hide.
- **The number over 500 is the "classic" one**, the most comparable with the literature, which has
  historically reported results on the `full` test set.
- **We don't lose the `sanitized` information: we reuse it as a label.** The `task_id`s that survive
  sanitization identify the "clean" problems. This allows us (during analysis) to split the 500 into
  two groups, **"clean"** (present in `sanitized`) and **"noisy"** (discarded/reworded), and to
  compare pass@1 across the two. The **gap** between the two groups *is* the measure of the cost
  that bad wording imposes on the model.

> ⚠️ **Interpretation caveat.** Some problems discarded by `sanitized` are not just "badly written":
> they are **under-specified** (the test demands something the prompt does not say). On those, not
> even a perfect model can always guess. So a lower pass@1 on the "noisy" group should be read as
> *performance on noisy prompts (some of which are unsolvable)*, not as "the model is worse".

---

### Two response formats: Completion and Insertion

DS-1000 inserts the model's solution in place of `[insert]` in the harness. Depending on where
`[insert]` is located, the expected solution takes **two** different **forms**; the pipeline
recognizes them by looking at the line preceding `[insert]`:

- **Completion**: `[insert]` is at module level (e.g. after `df = test_input`): the solution is a
  **snippet** that assigns the `result` variable.
- **Insertion**: `[insert]` is **inside** a function (`def f(df):`): the solution is the **body** of
  the function, must be **indented** and ends with `return`.

> **Indentation handled by the pipeline, not the model.** For the *Insertion* format we ask the
> model to write the body **at column 0** and the **executor** re-indents it at the right spot
> (`textwrap.indent`/`dedent`), also preserving the structure of any loops. This avoids the typical
> LLM flaw (first line at column 0, the rest indented) that produced an `IndentationError` even
> though the logic was correct.

---

### Execution and installed libraries

The verification program is `code_context` + the call to `test_execution(<solution>)` (for
*Matplotlib* problems the non-interactive `Agg` backend is forced). Because each library must be
**actually imported**, the pipeline **automatically detects** which are installed and **skips** the
problems of the others, **with no silent truncation**:

- At startup the panel lists the libraries being run and warns about the skipped ones, indicating
  how many problems they exclude and over how many pass@1 is computed (e.g. *955/1000*).
- With `--libraries` you can restrict to the subsets you care about (e.g.
  `--libraries Pandas Numpy Pytorch`, case-insensitive).

> **TensorFlow on Python 3.14.** At the moment TensorFlow **has no wheel for Python 3.14**
> (`pip install tensorflow` → *No matching distribution*), so its **45 problems** are not executable
> and are excluded: pass@1 is reported over **955/1000**. To run them you need a dedicated Python
> 3.12 environment.

> **Higher timeout (60 s).** Data-science imports in a fresh subprocess are slow: cold (cold cache /
> OneDrive / antivirus) `scipy`/`sklearn` can take 20-30 s just for the import. To avoid mistaking a
> slow import for a failure, DS-1000 uses a default timeout of **60 s** (overridable with
> `--timeout`), versus the 10 s of HumanEval/MBPP.

As with the other benchmarks, the dataset is saved locally in `Benchmark/ds1000/` on the first run
and read back from there afterward.

## Prompting: Directional Stimulus (DSP)

Prompting follows **Directional Stimulus Prompting** (Li et al., 2023): in addition to a fixed
instruction, **each problem** receives small *directional stimuli* extracted locally from the
statement (no additional API call, hence no waste).

- **Fixed instruction** (`DSP_INSTRUCTION` in `prompting.py`): guides the model to read the
  signature and docstring, study the examples and edge cases, then write the complete solution.
- **Per-problem stimuli** (`directional_stimulus`): the function signature, return type, function
  name (`entry_point`), doctest examples, edge-case keywords present in the docstring.
- **Format constraint** (`SYSTEM_PROMPT` in `config.py`): respond **with code only** (signature +
  imports), without prose or markdown blocks.

The generated code is then isolated by `code_extractor.py` (it also handles the case where the
model, disobeying, wraps its response in a ```` ```python ```` block).

> **MBPP has no signature.** Unlike HumanEval, an MBPP problem is just a natural-language description
> (`text`) + a list of `assert` statements (`test_list`): no signature and no `entry_point`. For
> this reason `build_prompt` recognizes the benchmark and, for MBPP, uses a dedicated prompt
> (`_build_prompt_mbpp`) that **infers the function's name and signature from the tests** and shows
> them to the model. In MBPP the tests *are* part of the specification (Austin et al., 2021):
> showing them is standard practice, not unfair help.

> **DS-1000 has two formats.** DS-1000 also uses a dedicated prompt (`_build_prompt_ds1000`): in
> addition to the target library, the *directional stimulus* indicates the **solution format**
> inferred from the scaffold: for *Completion* assign `result` using the data already provided
> (**without re-importing/redefining it**, otherwise the test's input would be overwritten); for
> *Insertion* write the function body at column 0 (the executor handles the indentation). The model
> is asked for its **natural** form (the snippet): it is **not** forced to imitate the
> "function-wrapper" form of the gold solution, because that would mean shaping the output to a
> metric rather than measuring it (and would increase the risk to pass@1).

## Evaluation Metrics

We keep **two** complementary metrics: a functional one (is the code correct?) and a structural one
(how much does it resemble the reference solution?).

### pass@1

The **pass@1** metric measures whether the **first** (and only) generated solution passes **all** of
the problem's unit tests. It is the primary metric: it captures real functional correctness. The
generated code is executed against the problem's tests in an isolated process, with a timeout.

> `executor.py` recognizes the benchmark: for HumanEval it assembles `imports + code + test +
> check(entry_point)`; for MBPP it assembles `test_setup_code + code + test_list` (the `assert`s,
> without `check`); for DS-1000 it assembles `code_context + test_execution(<solution>)`
> (re-indenting the solution when it must be inserted into a function body). Error classification
> (`AssertionError`, `SyntaxError`, `TimeoutError`, `ImportError`, …) and isolated execution with a
> timeout are shared.

---

### CodeBLEU

<table>
<tr>
<td width="120"><strong>📄 Paper</strong></td>
<td><a href="https://arxiv.org/abs/2009.10297">CodeBLEU: A Method for Automatic Evaluation of Code Synthesis</a> (Ren et al., 2020)</td>
</tr>
<tr>
<td><strong>📦 Library</strong></td>
<td><code>pip install codebleu</code> (see the tree-sitter note above)</td>
</tr>
</table>

**CodeBLEU** extends BLEU to code, combining four components:

| Component | Description |
|------------|-------------|
| **BLEU** | Standard BLEU over the code's n-grams |
| **BLEU-weighted** | BLEU weighted on the language's keywords (`def`, `return`, `if`, …) |
| **AST Match** | Similarity between the Abstract Syntax Trees |
| **Data-Flow Match** | Similarity in the data flow between variables |

We use uniform weights `(0.25, 0.25, 0.25, 0.25)`.

> **Fair comparison thanks to `codice_completo`.** The model generates the **complete function**
> (signature + imports + body). Comparing it with just the `canonical_solution` (body) would be
> unbalanced and would artificially lower the score. For this reason the CodeBLEU reference on
> HumanEval is `codice_completo = prompt + canonical_solution`, i.e. the complete reference function.
> **On MBPP** the reference is directly the dataset's `code` field (already a complete function). In
> both cases the **tests never enter** the CodeBLEU reference: the comparison is always
> code-vs-code.
>
> ⚠️ *Single reference.* Both HumanEval and MBPP provide **a single** canonical solution. CodeBLEU
> (like BLEU) was designed for comparisons against multiple references, so it should be read as *"how
> much it resembles the single reference solution"*: a correct but stylistically different solution
> can have a low CodeBLEU while still passing `pass@1`. The limitation is identical on HumanEval and
> MBPP, which therefore remain comparable to each other.
>
> ⚠️ **DS-1000: CodeBLEU is secondary.** In DS-1000 the `reference_code` is often in
> "function-wrapper" form (`def g(df): … ; result = g(df.copy())`), while the model's natural output
> is an inline snippet: the **form diverges by construction** and CodeBLEU comes out low *even when
> the solution is correct and passes the tests*. On top of that, on short snippets the AST/data-flow
> components degenerate. On DS-1000, therefore, **the reference metric is pass@1**; CodeBLEU should
> be read as a purely informative structural indicator, **not comparable** with HumanEval/MBPP.
>
> ⚠️ **Plot2Code and MultiPL-E.** *Plot2Code* computes CodeBLEU (generated code vs gold script) plus
> a composite **image similarity** (see *Benchmark: Plot2Code*). *MultiPL-E* **has no gold solution**
> → CodeBLEU is **not computable**: the metric is **`pass@1` only** (the records have `codebleu =
> None`).

### Which model solves the most problems on the first attempt?

After testing the three models on all five benchmarks, both the *Findings* table and the bar chart

on every benchmark, followed by **DeepSeek V4 Pro** and finally **Ministral 8B**.

The ordering mirrors the hierarchy of the architectures (frontier dense LLM, MoE, SLM), but the most
interesting fact is how much the gap varies from benchmark to benchmark. On HumanEval the three
models are relatively close (99.4%, 90.2%, 76.2%): these are classic interview-style problems, well
represented in the training data of any recent model, and the benchmark shows signs of saturation.
On DS-1000 the gap widens dramatically (81.9%, 61.6%, 35.9%): solving real data science problems
requires fine-grained knowledge of the library APIs, and this is exactly the kind of specialized
knowledge that grows with the model's capacity. In short, generalist benchmarks discriminate less
and less, while specialized ones still clearly separate the three architectures.

### Does correct code have to resemble the reference solution?

A natural question, looking at the two metrics together, is whether working code must necessarily be
identical (or very similar) to the benchmark's reference, or whether there are multiple paths to the
solution. The following scatterplots relate, for each problem, the CodeBLEU score to the outcome
(blue = solved, red = unsolved), for each model on the four benchmarks that have a reference
solution.

**Claude Opus 4.7 (LLM)**

<p align="center"><img src="plots/02_scatter_codebleu_claude-opus-4.7.png" alt="scatter CodeBLEU Claude" width="760"></p>

**DeepSeek V4 Pro (MoE)**

<p align="center"><img src="plots/02_scatter_codebleu_deepseek-v4-pro.png" alt="scatter CodeBLEU DeepSeek" width="760"></p>

**Ministral 8B (SLM)**

<p align="center"><img src="plots/02_scatter_codebleu_ministral-8b.png" alt="scatter CodeBLEU Ministral" width="760"></p>

The answer is no: the blue points (solved problems) are spread along the **entire** CodeBLEU axis,
including the band below 0.2. All three models regularly produce solutions that are correct but
structurally far from the canonical one. The benchmark's reference solution, in other words, is not
the only one possible: others exist, often more idiomatic, and the models find them.

A few concrete examples, one per model, make the point. Problem `HumanEval/35` asks for the maximum
of a list; the reference solution computes it explicitly with a `for` loop and an accumulator
variable:

```python
def max_element(l: list):
    m = l[0]
    for e in l:
        if e > m:
            m = e
    return m
```

Claude solves the same problem with the built-in function:

```python
def max_element(l: list):
    return max(l)
```

The generated code passes all the tests, but gets a CodeBLEU of just **0.11**: no loop, no
accumulator, almost no overlap of n-grams or syntactic structure with the reference. And yet it is
the more idiomatic of the two solutions.

The phenomenon is not limited to Claude: each model deviates from the reference in its own way.
DeepSeek, on problem `HumanEval/72` (`will_it_fly`: a list "flies" if it is a palindrome and the sum
of its elements does not exceed a maximum weight), is faced with a reference that checks the
palindrome with a two-index loop scanning the list from both ends:

```python
def will_it_fly(q,w):
    if sum(q) > w:
        return False

    i, j = 0, len(q)-1
    while i<j:
        if q[i] != q[j]:
            return False
        i+=1
        j-=1
    return True
```

and replaces it with a single line based on slicing (CodeBLEU **0.13**, tests passed):

```python
def will_it_fly(q,w):
    return q == q[::-1] and sum(q) <= w
```

Ministral, on problem `HumanEval/157` (`right_angle_triangle`: determine whether three sides form a
right triangle), goes in the opposite direction: where the reference solves it in one line by trying
the Pythagorean theorem on all three combinations of sides:

```python
def right_angle_triangle(a, b, c):
    return a*a == b*b + c*c or b*b == a*a + c*c or c*c == a*a + b*b
```

the model sorts the sides to test a single combination and even introduces a numerical tolerance
with `math.isclose` (CodeBLEU **0.14**, tests passed):

```python
import math

def right_angle_triangle(a, b, c):
    sides = sorted([a, b, c])
    return math.isclose(sides[0]**2 + sides[1]**2, sides[2]**2, rel_tol=1e-9)
```

Three models, three styles: Claude prefers the built-in, DeepSeek the one-liner with the language's
idioms, Ministral the more explicit and defensive version. In every case the code is functionally
correct but structurally far from the reference, with CodeBLEU between 0.11 and 0.14. Cases like
these explain why CodeBLEU should be read as a measure of **stylistic and structural similarity**,
never as a surrogate for correctness.

The two metrics, however, are not unrelated: in all the charts the red points (unsolved) cluster in
the lower part of the CodeBLEU axis, particularly on DS-1000 and Plot2Code. When the model really
does not know how to tackle a problem, its output tends to drift away from the reference both
semantically and in form.

be sacrificed.

2. **Syntax becomes a problem again only at the margins.** On MultiPL-E Ministral accumulates 306

   environment for the run to fail before even drawing. Ministral also shows a tail of runtime errors
   (`ValueError`, `TypeError`, `RuntimeError`) absent in the large models: the script gets written,
   but with internal inconsistencies (array dimensions, function arguments) that blow up at
   execution time.

The composition of the recoveries is instructive. About a hundred come from MultiPL-E and are
concentrated in the languages where Claude itself is least solid (25 in Go, 16 in R, 11 in Julia).
Another 34 come from Plot2Code, where in 28 cases Claude's failure was an `ImportError`: the script

concern Pandas and in 24 cases Claude's error was an `AssertionError`: an output close to, but not
identical to, the expected one.

The most emblematic case is the single recovery on HumanEval, problem `HumanEval/140`
(`fix_spaces`), whose specification reads:

> *Given a string text, replace all spaces in it with underscores, and if a string has more than 2
> consecutive spaces, then replace all consecutive spaces with -*

Claude generated a compact solution:

```python
import re
def fix_spaces(text):
    return re.sub(r' {2,}', '-', text).replace(' ', '_')
```

which, however, replaces sequences of **2 or more** spaces with `-`, while the specification asks for
"more than 2": a sequence of exactly two spaces should become `__`. DeepSeek, with a less elegant
solution, used the correct threshold (`{3,}`) and passed the tests.

The correct reading of these numbers is not that the small models "know" things Claude ignores. The
recurring mechanisms are three: ambiguous or tricky specifications, where different interpretations
are rewarded or punished by the tests (as in `fix_spaces`); simpler and more conservative solutions,
which avoid the errors that a more sophisticated solution runs into (the `ImportError`s of
Plot2Code); and the intrinsic variance of the single sample, since pass@1 grants a single attempt.
There remains, however, a relevant practical fact: no model strictly dominates the others, and a
system that combined several models would have real recovery margins.

### Plot2Code: how much does the generated figure resemble the reference?

For Plot2Code, in addition to the rendering (pass@1) and CodeBLEU, a composite image similarity
`img_visual` (SSIM + color histogram + OCR) is computed between the generated PNG and the reference
one. The average similarity follows the same hierarchy as pass@1: **0.69** for Claude, **0.63** for
DeepSeek, **0.55** for Ministral. No model reproduces the figures faithfully (a value close to 1
would be needed): the textual description leaves freedom over colors, proportions and details that
the perceptual metric records as a difference.

The following scatterplot cross-references the two similarities, that of the code and that of the
image. For the strong models the two measures are correlated (Claude r ≈ 0.61, DeepSeek r ≈ 0.49): a
code closer to the reference script tends to produce a more similar figure. For the small model the
correlation is nearly zero (Ministral r ≈ 0.03): its code can resemble the reference and still draw a
different figure, or vice versa. To evaluate chart generation, therefore, the decisive metric is the
one on the image: CodeBLEU alone is not enough.

### Conclusions

The overall picture can be summarized in five points.

1. **The architecture hierarchy is real, but not uniform.** The frontier dense LLM wins on all
   benchmarks, but the gap depends on the task: minimal on generalist problems (HumanEval), maximal
   where specialized knowledge is needed (DS-1000). The MoE, with about 13 billion active parameters
   and a per-token price more than an order of magnitude lower, remains surprisingly competitive on
   the standard benchmarks; the SLM holds up on simple problems but collapses where depth is needed
   (35.9% on DS-1000, 25.9% on MultiPL-E's Bash).
2. **Correctness and similarity to the reference are distinct properties.** The models regularly
   solve problems with solutions structurally far from the canonical one (the `max_element` case):
   CodeBLEU is a useful complementary stylistic signal, but pass@1 remains the only reliable measure
   of correctness. For visualization, where the code is just a means, a perceptual metric on the
   produced image is needed.
3. **The bottleneck is semantics, not syntax.** The dominant error is everywhere the
   `AssertionError`: valid code that does the wrong thing. Syntax becomes an obstacle again only for
   the small models on low-resource languages.
4. **No model dominates in the strict sense.** 180 problems failed by Claude were solved by weaker
   models, typically due to different interpretations of ambiguous specifications or the choice of
   simpler solutions. With a single attempt available, a share of variance is inevitable, and it is a
   concrete argument in favor of multi-model strategies in real systems.
5. **Multilingual coverage mirrors the training data.** All models excel at JavaScript, TypeScript
   and Java and struggle with Bash, R and Julia; the gap between the large and small models widens
   precisely on the rare languages, where the small one even stops producing syntactically valid
   code.

## Pause and Resume

Execution can be interrupted and resumed without re-paying for queries already spent:

- **Ctrl+C** once → **clean** stop: the problem in progress is completed and saved, then the run
  stops. **Ctrl+C** a second time → forced stop.
- Progress is saved problem-by-problem in `results/<benchmark>/<model>.jsonl` (append-only, with a
  flush per line): it is the source of truth for resuming, robust even to a forced stop.
- On **relaunching** the same command, the problems already done are **skipped**. API errors
  (infrastructure) **do not** count as "done" and are **retried**.
- `--fresh` ignores the checkpoint and restarts from scratch.

## Error Analysis

When a problem does not pass (pass@1 = 0), the failure is **categorized**, so as to understand *how*
the model goes wrong, not just *how much*. The error distribution is one of the three tables in the
final summary.

### Categories (Python: HumanEval, MBPP, DS-1000, Plot2Code)

| Category | Description | Example |
|-----------|-------------|---------|
| **SyntaxError** | Code with Python syntax errors | Unclosed parentheses, wrong indentation |
| **IndentationError** | Invalid indentation | Mix of tabs and spaces, misaligned blocks |
| **NameError** | Reference to undefined variables/functions | Use of a variable before assignment |
| **TypeError** | Operation on an incompatible type | Sum between a string and an integer |
| **ValueError** | Invalid value for the operation | `int("abc")` |
| **IndexError** | Access to an out-of-range index | List of 3 elements, access to index 5 |
| **KeyError** | Key not present in the dictionary | `dict["nonexistent_key"]` |
| **AttributeError** | Non-existent attribute/method | Call to a non-existent method |
| **AssertionError** | Test failed (wrong logic) | Output consistent in structure but wrong value |
| **TimeoutError** | Execution too long | Infinite loop, inefficient algorithm |
| **APIError** | Infrastructure error in the API call | Does not count as "done": it is retried on resume |

---

### Categories for non-Python languages (MultiPL-E)

For languages other than Python there is **no Python traceback**: the executor infers the category
from the runtime/compiler **error message** (`_classify_multipl_e` in `executor.py`), mapped back to
the **same categories** as the table above for a homogeneous comparison across languages. The checks
go from the most specific to the most generic, with a `RuntimeError` fallback.

| Category | How it is recognized (multilingual signatures) |
|-----------|--------------------------------------------------|
| **SyntaxError** | **Compilation** fails (Java/C++/Rust/C#/…) or parsing does: `syntax error`, `parse error`, `unexpected token`, `unterminated`, `expected …` |
| **AssertionError** | A test fails: `assertion failed`, `panicked` (Rust), `stopifnot` (R), `not ok` (Perl/TAP), `--- FAIL` (Go) |
| **ImportError** | Missing module/library: `cannot find module` (JS), `can't locate` (Perl), `there is no package` (R) |
| **NameError** | Undefined reference: `is not defined` (JS), `command not found` (shell), `undefined subroutine` (Perl), `nil value` (Lua) |
| **TypeError** | Incompatible type: `is not a function`, `cannot read propert…`, `non-numeric argument` |
| **IndexError** | Out-of-range access: `index out of`, `out of range`, `subscript out of bounds` |
| **ValueError** | Invalid value or division by zero: `division by zero`, `divide by zero` |
| **RuntimeError** | Any other runtime crash (fallback category) |
| **RuntimeMissing** | **The language's runtime/compiler is not installed** → it is not a model error, and MultiPL-E **does not even generate** for that language (see *Benchmark: MultiPL-E*) |

> The compiled languages (Java, C++, Rust, C#, …) pass `compile_error=True` when the build fails: in
> that case the category is always **SyntaxError**, because the code did not even reach execution.
